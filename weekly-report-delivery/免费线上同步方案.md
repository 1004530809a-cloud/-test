# 周报与低毛利反馈免费线上同步方案

## 1. 结论

推荐主方案：**飞书多维表格 + 现有网页只读展示/导出**

原因：

- 更贴近当前 Excel 使用习惯，业务同学上手成本最低。
- 支持多人同时编辑，天然解决本地文件反复传版本的问题。
- 低毛利反馈可直接在线批量填写，不必继续依赖本地 `file://` 文件。
- 可以按“月份”保留历史表，也可以按“周报批次”保留快照。
- 后续若还想保留现有网页周报，只需要把网页的数据源从本地 JSON 改成“在线拉取一份已整理数据”。

不推荐当前阶段直接上自建数据库：

- 你现在的核心痛点是“同步、留痕、批量填写”，不是复杂权限或高并发。
- 如果立刻切到 Supabase/自建后端，技术成本会明显高于业务收益。
- 现阶段更适合先把流程在线化，再决定是否需要系统化升级。

## 2. 推荐架构

### 2.1 数据分层

把数据拆成 4 层：

1. `订单原始导入层`
   - 每周导入“确认订单”“执行订单”原始数据。
   - 保留原始字段，不做人工修改。

2. `低毛利反馈层`
   - 从执行订单中筛出 `预估订单毛利率 < 14%` 的订单。
   - 在线填写“原因”“备注”“责任人”“处理状态”。

3. `周报汇总层`
   - 按你已确认口径自动汇总 SZA / 非SZA 指标。
   - 非SZA TOP5 单独摘录展示。

4. `历史快照层`
   - 每次生成周报时，固化一份“周报快照”。
   - 避免后续源数据被改动后，历史周报口径漂移。

### 2.2 表设计

建议在一个飞书多维表格里建这些表：

- `确认订单_原始导入`
- `执行订单_原始导入`
- `低毛利反馈_当月`
- `低毛利原因字典`
- `周报快照`
- `周报风险事项`

其中关键字段建议如下。

`执行订单_原始导入`

- `周报周期`
- `采购订单编号`
- `广告主名称`
- `是否SZA`
- `SPU类目`
- `执行价(含税)`
- `预估订单毛利额`
- `预估订单毛利率`
- `EV负责人`
- `是否低毛利`
- `导入批次`
- `导入时间`

`低毛利反馈_当月`

- `月份`
- `周报周期`
- `采购订单编号`
- `广告主名称`
- `SPU类目`
- `执行价(含税)`
- `预估订单毛利率`
- `原因`
- `备注`
- `反馈人`
- `反馈时间`
- `处理状态`
- `来源导入批次`

`周报快照`

- `周报周期`
- `生成时间`
- `生成人`
- `SZA成交金额`
- `非SZA成交金额`
- `SZA预估毛利率`
- `非SZA预估毛利率`
- `低毛利订单数`
- `低毛利成交金额`
- `已填写原因`
- `未填写原因`
- `周报JSON`

## 3. 业务规则落地

你当前已经确认的规则，建议直接固化为字段公式或导入脚本逻辑：

- `是否SZA = 广告主名称 包含 "华为"`
- 先过滤 `SPU类目` 含“汽车”的数据
- 成交金额取执行订单 `执行价(含税)`
- 订单数量取确认订单
- 非SZA核心指标按全部非华为订单汇总
- 非SZA负责人取执行订单 `EV` 中文名
- 低毛利订单取执行订单 `预估订单毛利率 < 14%`

注意：

- “是否SZA”“是否低毛利”都建议做成显式字段，不要每次靠人工理解。
- 周报展示层只能读“处理后数据”，不要直接从原始表临时拼。

## 4. 低毛利反馈在线化方案

### 4.1 填写方式

把现在 Excel 的“原因下拉”迁到 `低毛利原因字典` 表维护，再在 `低毛利反馈_当月` 中引用。

这样可以做到：

- 多人同时在线填写
- 原因字典统一维护
- 每条反馈都能看到是谁、什么时候修改
- 月度数据天然留存

### 4.2 批量反馈

批量反馈建议保留两种入口：

1. 表格视图批量编辑
   - 适合运营同学直接改多行原因。

2. 表单视图补充说明
   - 适合零散追单时从手机临时补录。

### 4.3 月度历史

建议不要继续让“低毛利反馈表.xlsx”作为主数据源。

改成：

- 当月在线表为主
- 需要归档时，按月导出一份标准 Excel 作为备份
- 导出文件名固定为 `低毛利反馈表_YYYYMM.xlsx`

## 5. 现有网页怎么接入

当前项目的真实瓶颈有两个：

- `weekly-report.js` 仍依赖本地缓存和本地上传
- `build_weekly_report_json.py` 输出的是本地 JSON 和本地 Excel

所以建议分两步走。

### 第一步：先把“数据录入”在线化

- 原始订单导入到在线表
- 低毛利反馈在线填写
- 周报网页暂时继续读取导出的 JSON

这样改动最小，但已经解决 80% 的协同问题。

### 第二步：再把“网页数据源”在线化

- 增加一个“在线汇总导出”脚本
- 脚本从在线表拉取数据
- 按现有口径生成 `weekly-report-data.generated.json`
- 网页继续渲染 JSON，不需要大改 UI

这样你现在的前端页面、打印、PDF 导出能力都能保住。

## 6. 历史留存建议

历史留存不要只靠版本历史，建议同时保留两份：

1. `在线表历史`
   - 适合查“谁改了什么”。

2. `周报快照`
   - 适合查“某一周最终对外汇报的结果是什么”。

这两者不能互相替代。

如果只看在线表历史，会很难快速还原当周最终汇报口径；
如果只留周报快照，又看不到中途是谁修改了低毛利原因。

## 7. 导出格式固定建议

“低毛利反馈导出表”建议固定成一个模板，而不是每次动态拼接。

建议规则：

- 第一行表头完全对齐模板 sheet `3月影响毛利订单反馈`
- 黄色表头字段顺序固定
- `原因` 列始终放最后
- 数据区统一边框、字号、行高
- 文件名固定
- 同一月份重复导出时覆盖同名文件或追加版本号

这样能减少：

- 模板样式漂移
- WPS 打开后的格式错位
- 同事自己另存成多个版本

## 8. 建议实施顺序

### 本周能落地

1. 先定在线主载体为飞书多维表格
2. 建 6 张基础表
3. 把“低毛利原因字典”迁过去
4. 先手工导入一周执行订单验证流程
5. 用在线表完成一次真实低毛利反馈

### 下周再做

1. 写导入脚本，把 Excel 自动导入在线表
2. 写汇总脚本，输出周报 JSON
3. 网页改成读取在线生成的 JSON
4. 固化低毛利反馈导出模板样式

## 9. 风险与对应处理

### 风险 1：表头变动导致导入失败

处理：

- 继续按“表头名”匹配，不按列字母取值
- 导入前先做字段校验并给出缺失提示

### 风险 2：历史数据被在线修改后影响旧周报

处理：

- 每周生成周报后写入 `周报快照`
- 快照只追加，不回写

### 风险 3：原因字典变动后影响统计口径

处理：

- 原因字典单独维护
- 已写入历史记录的原因值不反算

### 风险 4：WPS 与 Excel 样式兼容

处理：

- 在线表作为主数据源
- Excel 只作为导出交付格式
- 样式模板固定，不再让人工二次改版

## 10. 最终建议

当前阶段最优解不是“把现有本地网页直接改成多人实时编辑网页”，而是：

**先把数据源在线化，再让网页变成稳定的展示层。**

这样成本最低，也最符合你现在已经跑起来的流程。

如果后续数据量、权限复杂度、自动化要求再上升，再考虑把在线表升级成数据库后端。
